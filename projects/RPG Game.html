Introduction
Gloomhaven is a role playing board game. It is Dungeons and Dragonesque in theme but does not have any dialog exchanges. The story is static between play throughs, but it has altering paths, difficulties, and characters.

Bitmap

The majority of data for current state of the game is stored in a bitmap. It is a X by X 2 dimensional array. Each bit in the map can be a different character that describes a player, enemy, doorway, etc. Each game level can be loaded from a textfile like the one to the right. The text file is run through a file stream that can then send each character from the text file to a respective bitmap position. An important note is that the game is made of hexagons. This means that every other row is shifted half a hex over.
This video demos the capabilities of the bitmap. The starting positions are all described by the same text file. However, as play progresses the bitmap can update. There are some cool player friendly additions I made that I will jsut gloss over. The way the game tracks the mouse only allows for the player to select open hexagons. In other words, the player cannot move into an enemy or a block outside of the dungeon even though these hexagons exist in the bitmap. Also notice that the red hex that highlights the hex your mouse if hovering will automatically snap to the hexagon closest to the mouse. This way you can see which direction your mouse is if its off screen. It also lets you have more space for clicking. The next topic I will dive into that is the way the player moves around the bitmap.

Movement

In the video, you may notice that player will avoid walls and enemies automatically when pathing. This was on purpose and was quite difficult. The way the algorithm works in principle is similar to the way water would act if it was poured over a maze. Starting from the player different pathes are erected and tracked. Look to the image slides. Pretend the red hexagon is where the movement starts and the blue hexagon is where it is going to finish. Although normally, the wizard would be the starts and the mouse clicked red hex would be the end. I just felt this made the algorithm easier to visualize. As the arrows radiate outwards, they do not overlap. The algorithm will track each arrow and store all arrows in a path. On each iteration, none of the hexagons are revisited. Once the final hex is reached, the algorithm stops and checks to make sure that the quickest pathes is returned. Sometimes the algorithm will double up on a hexagon in a single iteration, but that is to solve for a possibility of not finding the quickest path but one from the quickest path. It still essenstially works the same.
